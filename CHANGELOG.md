# Changelog

All notable changes to the OpenClaw Middleware Suite are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- Model Routing: per-profile tier configurations. The store ([ModelRoutingPolicyData](src/middlewares/model-routing/storage/ModelRoutingPolicyStore.ts)) now carries `tierOverridesByProfile: Partial<Record<RoutingProfile, Partial<Record<Tier, TierModelConfig>>>>` — each of the three profiles (eco / premium / agentic) keeps its own tier→model map independently. Editing one profile's tiers in the dashboard or via `sai router tiers --profile <p> --set "<TIER> <model>"` no longer overwrites the others' saved configuration. The runtime config gains a `ModelRoutingConfig.tiersByProfile` field; [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) resolves the request's profile, then looks up `tiersByProfile[profile]` for tier→model resolution. Defaults flow from `PROFILE_CONFIGS` (static) at runtime; the init wizard precomputes `buildProfileFromDiscovered` defaults for each profile and persists them so every profile has an explicit configuration on a fresh install.
- Shared utility: extracted OpenClaw session-startup detection into [src/shared/session-detection.ts](src/shared/session-detection.ts) — a single `SESSION_STARTUP_REGEX` / `isSessionStartupMessage(text)` source of truth consumed by both Model Routing (for the `session_startup` scoring override) and Context Editing (for the first-message exclusion in `filterMessagesForICC`). The regex covers all three current OpenClaw reset-prompt shapes (BASE, BOOTSTRAP_PENDING, BOOTSTRAP_LIMITED) and is intentionally NOT anchored to start-of-string so it still matches when openclaw >= 2026.4.27 prepends a `[Startup context loaded by runtime]` daily-memory prelude to the same user-role body. Future OpenClaw prompt drift fixes one regex; both middlewares pick it up. New tests: [test/shared/session-detection.test.mjs](test/shared/session-detection.test.mjs) (13 pure-utility), [test/model-routing/session-startup-override.test.mjs](test/model-routing/session-startup-override.test.mjs) (7 `scoreRequest` integration), [test/context-editing/session-startup-filter.test.mjs](test/context-editing/session-startup-filter.test.mjs) (11 `filterMessagesForICC` integration).
- ICC extraction signaling end-to-end. New shared utility [src/shared/icc-detection.ts](src/shared/icc-detection.ts) defines `ICC_EXTRACTION_MARKER = '[SAI:ICC_EXTRACTION]'`, `isIccExtractionCall(text)`, and `stripIccMarker(text)` — the contract between Context Editing's compaction-extraction calls and Model Routing's scoring override. **CE side**: [ContextCurator.ts](src/middlewares/context-editing/ContextCurator.ts) `composeExtractionPrompt` prepends the marker only when the resolved target is `sai-router/*` (i.e., the call is actually going through MR); for direct providers (`anthropic/...`, `openai/...`) the marker is omitted to avoid LLM-prompt noise on calls MR will never see. **MR side**: a new `icc_extraction` branch in [overrides.ts](src/middlewares/model-routing/scoring/overrides.ts) `checkOverrides` runs **before** `session_startup`, returns `{ tier: 'SIMPLE', score: -0.5, confidence: 1.0, reason: 'icc_extraction' }`, and is added to the early-return list at [scorer.ts](src/middlewares/model-routing/scoring/scorer.ts) so `applyToolFloor` / `applyStructuredOutputFloor` cannot escalate the tier (the literal "schema" keyword in the ICC prompt would otherwise trip the structured-output floor). After scoring, [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) calls `stripIccMarker(...)` on every user-role `body.messages` entry (handles both `content: string` and `content: [{type: 'text', text}]` shapes) so the upstream LLM provider never sees the marker. Net effect: a user setting `agents.defaults.compaction.model = sai-router/eco` lands compaction calls on the eco profile's SIMPLE tier deterministically, regardless of the transcript content the scorer would otherwise see. Added a new value `'icc_extraction'` to the `ScoringReason` union in [types.ts](src/middlewares/model-routing/types.ts).
- Cost source attribution + per-source daily budgets. [cost-tracker.ts](src/middlewares/model-routing/storage/cost-tracker.ts) now accepts an optional `source: CostSource` (`'chat' | 'icc' | (string & {})`) on every `record()` call; events without a source default to `'chat'` so legacy callers stay attributed correctly. `DailyCost` gained a `bySource: Record<CostSource, SourceCostEntry>` map with per-source `costUsd` / `requestCount` and independent `warnFired` / `criticalFired` latches. `CostAlertConfig.budgets` is a new optional field of shape `Partial<Record<CostSource, { dailyWarn?: number; dailyCritical?: number }>>` — sources without an entry are still tracked but not alerted on individually; per-source alerts log with a tagged prefix (`WARNING [icc]: ...`) for grep-ability. [DEFAULT_COST_ALERT_CONFIG](src/middlewares/model-routing/storage/cost-tracker.ts) ships with `budgets: { icc: { dailyWarn: 1.0, dailyCritical: 5.0 } }` so a runaway compaction loop is visible without affecting normal chat alerting. [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) derives `costSource` from `scoringResult.reason === 'icc_extraction'` once after `FINAL_TIER` and threads it through all three `costTracker.record()` call sites (the SSE-converter completion handler, the fallback-executor stream callback, and the terminal cost record). Per-source alerts fire **independently** of the existing aggregate `warnThresholdUsd` / `criticalThresholdUsd` — a chat day can trip the aggregate warn while an `icc` day can simultaneously trip its own per-source warn on the same call. New tests in [test/model-routing/cost-tracker-source.test.mjs](test/model-routing/cost-tracker-source.test.mjs) cover bucketing, default-to-chat, per-source warn/critical firing, latching, and independence from aggregate alerts.
- Shared OpenClaw envelope stripper. New utility [src/shared/openclaw-envelope.ts](src/shared/openclaw-envelope.ts) exports `LEADING_TIMESTAMP_PREFIX_RE`, `INBOUND_META_SENTINELS`, and `stripOpenClawEnvelope(text)` — the contract for removing the AI-facing scaffolding OpenClaw prepends to every user turn. Two pieces of envelope are handled: the `[Day YYYY-MM-DD HH:MM TZ]` timestamp prefix (`injectTimestamp()` in openclaw's `gateway/server-methods/agent-timestamp.ts`) and any sentinel-delimited fenced-JSON metadata block (`buildInboundUserContextPrefix()` in openclaw's `auto-reply/reply/inbound-meta.ts`). The regex and sentinel list are kept verbatim from openclaw's official `stripInboundMetadata` in `auto-reply/reply/strip-inbound-meta.ts` so a future drift in envelope format breaks one regex in one place. The function is a zero-allocation no-op when neither pattern is present, so it adds no overhead to non-OpenClaw callers, and the strip is defensive — a sentinel header without a fenced-JSON body is treated as user content (no false strips on legit user text that happens to contain a sentinel string). Covers all six current sentinels: `Conversation info`, `Sender`, `Thread starter`, `Replied message`, `Forwarded message context`, `Chat history since last reply`. Both 2026.4.11 (envelope inlined into user-role body) and 2026.4.27 (sender block split into a separate `role: "custom"` chat-completion message) are handled. New tests in [test/shared/openclaw-envelope.test.mjs](test/shared/openclaw-envelope.test.mjs) lock both versions' shapes in as fixtures.
- Dashboard (Model Routing): three new surfaces on the page header / Overview tab. **Refresh button** — top-right of the page header, in line with the Overview / Config / Logs tab strip. Hoisted the data-loading from `useEffect` into a reusable `loadAll()` so the same fetch fan-out powers both initial mount and explicit refresh. Spins via a new `@keyframes sai-spin` in [components.css](src/dashboard/styles/components.css) while in flight, label switches to `Refreshing…`, button is disabled while either initial load or refresh is pending ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)). **Reason + Confidence columns** in "Recent Routing Decisions" — both fields are already on `RoutingAuditEntry` (the audit log writes them per request); surfacing them here brings the dashboard table in sync with the `Tier / Model / Score / Conf / Reason` columns the `sai router stats` CLI prints, so users can correlate routing decisions across both surfaces. **Cost Sources Today** section — two cards (Chat / ICC Compaction) under the cost chart, always rendered (including on fresh enable when both spends are 0) so the section is discoverable from day one. Each card shows today's spend, request count, the source's description, and a horizontal progress bar capped at the source's critical threshold (or warn if no critical), filled green / amber / red depending on which threshold the spend has crossed. "Warn $X / Critical $Y" labels under the bar; falls back to "No per-source budget configured." when `costAlerts.budgets[source]` is unset. Two new helpers (`computeTodayBySource`, `SourceCostCard`) live at the bottom of the page module.

### Changed

- Model Routing: there is no user-facing "default profile" concept anymore. All three profiles are always exposed in the OpenClaw model picker; users select one per request rather than configuring a single global default. The dashboard's "Default Profile" dropdown was relabeled "Editing profile" and now scopes tier writes to the chosen profile only — submitting the form sends `{ profile, tierOverrides, sessionPinningEnabled, providerCacheEnabled }` to `PUT /api/routing/tiers`, which nests `tierOverrides` under `tierOverridesByProfile[profile]` ([dashboard-api.ts](src/shared/server/dashboard-api.ts), [ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)). Switching the editing profile remounts the form (`key={editingProfile}`) so tier dropdowns re-seed from that profile's saved values. The internal `defaultProfile` field on `ModelRoutingConfig` is preserved as the **absolute fallback** when a request can't be matched to a profile (no `sai-router/*` model id, no `X-Router-Profile` header) — programmatic consumers can still pass `mr.initialize({ defaultProfile: 'premium' })` to control that. The proxy's `resolveProfile` falls back to `'eco'` if `defaultProfile` is somehow not a valid profile ([proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts)).
- Model Routing: `sai router tiers` now requires `--profile <eco|premium|agentic>` to scope an edit. `sai router tiers --profile premium --set "COMPLEX claude-opus-4-6"` writes only the premium profile's COMPLEX tier; the eco and agentic profiles are untouched. Without `--profile` the command prints the tier mappings for all three profiles, marking each tier as `(configured)` or `(default)` ([cli/tiers.ts](src/middlewares/model-routing/cli/tiers.ts)).
- Model Routing: `sai init` interactive flow reworked. The wizard now skips the "select one profile" prompt and instead asks "Would you like to customize tier mappings for any profile?" When yes, the user picks any subset of `eco / premium / agentic` from a multiselect and walks through tier customization for each chosen profile. Profiles the user doesn't customize get `buildProfileFromDiscovered` defaults persisted explicitly so every profile slot exists in storage on a fresh install. The universal-fallback step now applies the chosen fallback across every tier of every profile ([cli/init.ts](src/middlewares/model-routing/cli/init.ts)).
- Model Routing: `buildConfig` ([index.ts](src/middlewares/model-routing/index.ts)) deep-clones `tiersByProfile` from the default before applying overrides so cross-call mutations don't leak into the module-level constant.
- Model Routing: renamed the four routing model display names from `"Auto (Smart Router — Balanced)"` / `"Eco (Smart Router — Cost Optimized)"` / `"Premium (Smart Router — Best Quality)"` / `"Agentic (Smart Router — Tool Optimized)"` to `"sai-router/auto"` / `"sai-router/eco"` / `"sai-router/premium"` / `"sai-router/agentic"` in [router-provider.ts](src/middlewares/model-routing/router-provider.ts). The OpenClaw `/model` UI dropdown renders `entry.name`, while `openclaw configure` labels options with the raw `provider/id` key — so the two surfaces showed different strings for the same routing models. Aligning `name` to the key form makes both surfaces match. Model `id`s are unchanged, so the routing proxy and tier resolution are unaffected. Existing users will pick up the new names automatically by toggling the middleware off and on in the dashboard (the disable cleanup strips `sai-router` entries from openclaw.json, and the re-enable persist-defaults handler runs the full init which re-injects fresh model definitions via `injectModelsConfig`).
- Plugin runtime: prefer the new `runtime.config.current()` / `runtime.config.replaceConfigFile()` APIs (introduced in OpenClaw 2026.4.27) over the now-deprecated `loadConfig()` / `writeConfigFile()` helpers (compat code `runtime-config-load-write`). [config-manager.ts](src/plugin/config-manager.ts) feature-detects at every call site and falls back to the deprecated APIs when the new ones aren't present, so the suite continues to support both 2026.4.11 – 2026.4.26 (old APIs only) and 2026.4.27+ (both APIs, new preferred) — matching the package's `peerDependencies: openclaw >=2026.4.11`. The `OpenClawRuntime` interface in [plugin/index.ts](src/plugin/index.ts) declares all four methods optional with version-tagged JSDoc. Writes use `replaceConfigFile({ nextConfig, afterWrite: { mode: "auto" } })`; the `"auto"` policy delegates reload-vs-restart to the gateway's reload planner, matching the old `writeConfigFile` behavior. `loadOpenClawConfig` / `saveOpenClawConfig` keep the same external signatures, so all callers — Model Routing's `injectModelsConfig` / `cli/init.ts`, Context Editing's `cli/init.ts` / `pruning.ts` / `model.ts`, the disable-cleanup path in [shared/storage/cleanup.ts](src/shared/storage/cleanup.ts), `stageOpenClawWrites` / `flushToOpenClaw` in [shared/server/openclaw-sync.ts](src/shared/server/openclaw-sync.ts), and the toggle CLI in [shared/cli/toggle.ts](src/shared/cli/toggle.ts) — pick up the runtime API transparently. New tests in [test/plugin/config-manager-runtime-compat.test.mjs](test/plugin/config-manager-runtime-compat.test.mjs) verify both populations are routed correctly and that the old-API path is never invoked when the new APIs exist.
- Context Editing: ICC extraction now honors `agents.defaults.compaction.model` and falls back through the agent primary model on failure. Both extraction paths in [ContextCurator.ts](src/middlewares/context-editing/ContextCurator.ts) (`extractViaLLM` and `extractViaCustomPrompt`) iterate a target list returned by the new `resolveExtractionTargets(api)` helper — `compaction.model` first, `model.primary` second, deduped — and try each in order until one succeeds. Previously only `model.primary` was read, which silently routed every compaction call through the user's chat model regardless of how the user had configured `sai context-editing model --set <model>` or the dashboard's compaction-model field. The cascade is now `compaction.model → model.primary → regex extraction`; a misconfigured compaction model (wrong id, bad API key, provider down) silently downgrades to the primary model instead of degrading all the way to regex. JSON-parse errors do **not** trigger fallback (they're upstream-format issues, not target-failure issues) and surface as `null` immediately. The marker decision is per-attempt — when the first target is `sai-router/*` and the second target is a direct provider, the marker is included on attempt 1 and dropped on attempt 2 so the direct-provider call doesn't carry unexplained noise. Each ICC call site went from ~14 lines of duplicated config-reading + slash-splitting + provider/model assembly to a single `resolveExtractionTargets(api)` call.
- Dashboard (Model Routing): page-layout cleanups on the Config tab. The **Editing profile** selector now uses the same `.config-field` / `.config-field-row` grid layout as every form field below it — info on the left (label + one-line description), the dropdown right-aligned in the control column. Description trimmed to a single sentence ("Pick which routing profile's tier mappings you want to edit.") since the full multi-sentence rationale was rendered between the label and the control and visually competed with the dropdown. The selector still lives outside `<ConfigForm>` so `key={editingProfile}` continues to remount the form on profile switch. The **four tier dropdowns** (SIMPLE / STANDARD / COMPLEX / REASONING) are now visually nested under the profile selector via `indent: true` on their `FormField` definitions, reusing the existing `.config-field--nested` styling that Context Editing's config page uses for `pruning.maxIdleHours` under "Inactive Session Pruning". Universal Fallback / Session Pinning / Provider Prompt Caching stay un-indented because they're profile-level settings, not tier mappings ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)).

### Removed

- **Breaking (programmatic API)** — Model Routing: removed the flat `tiers` field from `ModelRoutingConfig` and the legacy `rawConfig.tiers` shim in `buildConfig`. Programmatic consumers that previously read `mr.getConfig().tiers[result.tier]` or passed `mr.initialize({ tiers: {...} })` must migrate to the per-profile shape: `mr.getConfig().tiersByProfile[profile][result.tier]` for reads, and `mr.initialize({ tiersByProfile: { eco: {...}, premium: {...}, agentic: {...} } })` for writes (any subset of the three profiles is fine; missing profiles inherit `PROFILE_CONFIGS` defaults). The runtime `config.tiers` mirror is gone, the `DEFAULT_MODEL_ROUTING_CONFIG.tiers` field is gone, and the proxy's observability log under `/v1/status` now emits `tiersByProfile` instead. Updated the README and `pickTier` / `updateConfig` docstrings in [middlewares/model-routing/index.ts](src/middlewares/model-routing/index.ts) accordingly.
- Model Routing: dropped the `--set-profile` CLI option from `sai router config`. The "default profile" concept it edited no longer exists at the user-facing level — all three profiles (eco / premium / agentic) are always available in the OpenClaw model picker, and per-request profile selection happens via the model id. Programmatic consumers can still set the absolute fallback via `mr.initialize({ defaultProfile })`. Also removed the legacy flat `tierOverrides` field from `ModelRoutingPolicyData` — old configs that have it persisted are simply ignored; users can re-run `sai init` or edit tiers via the dashboard / `sai router tiers --profile <p>` to populate `tierOverridesByProfile`.
- Model Routing: dropped the `auto` routing profile. The dashboard / CLI / OpenClaw dropdown now expose only `eco`, `premium`, and `agentic`, and the new default profile is `eco`. The `auto` profile previously meant "balanced — defer to whatever tier assignments are in `DEFAULT_MODEL_ROUTING_CONFIG.tiers`," which collided conceptually with "auto routing" itself (the scoring engine's per-request tier selection) and confused users. Removed across [selection/profiles.ts](src/middlewares/model-routing/selection/profiles.ts) (type, `VALID_PROFILES`, `PROFILE_CONFIGS`, `PROFILE_DESCRIPTIONS`, the `generateProfileFromDiscovered` short-circuit, and the `buildCatalogAuto` / `discoveredAuto` helpers), [config.ts](src/middlewares/model-routing/config.ts) (`defaultProfile: 'auto'` → `'eco'`), [router-provider.ts](src/middlewares/model-routing/router-provider.ts) (the `id: 'auto'` entry from `ROUTING_MODELS`), [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) (`ROUTING_META_MODELS` set, the `'auto'` literal special-case in `resolveProfile`, and the four outbound `X-Router-Profile` header guards), [router-config-inject.ts](src/middlewares/model-routing/router-config-inject.ts) (`ALLOWLIST_MODELS`), [cli/config.ts](src/middlewares/model-routing/cli/config.ts), [cli/init.ts](src/middlewares/model-routing/cli/init.ts), and [dashboard/pages/ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx). Legacy clients still sending `model: "sai-router/auto"` fall through to header-then-`defaultProfile` resolution in the proxy and continue to route — the literal id is no longer special-cased but its old behavior (use `defaultProfile`) is preserved by the existing fallback path.

### Fixed

- Model Routing: session-startup detection now matches the OpenClaw 2026.4.27 reset-prompt shapes. `BARE_SESSION_RESET_PROMPT_BASE` was reworded from `"Run your Session Startup sequence"` (≤ 2026.4.11) to `"Execute your Session Startup sequence now"` (≥ 2026.4.27), and two new variants were introduced — `BARE_SESSION_RESET_PROMPT_BOOTSTRAP_PENDING` and `BARE_SESSION_RESET_PROMPT_BOOTSTRAP_LIMITED` — that begin `"… /new or /reset while bootstrap is still pending …"` instead of ending the sentence after `/reset`. Additionally, when a workspace has any `memory/YYYY-MM-DD.md` daily-memory files, the gateway prepends a `[Startup context loaded by runtime]` prelude inside the same user-role body, so the reset prompt no longer sits at position 0. The previous `^A new session was started via /new or /reset\.\s*Run your Session Startup sequence` regex broke on all three counts: every `/new <text>` turn fell through to the keyword scorer with a ~3.4k-char prelude + reset-prompt blob attached, classifying as STANDARD or COMPLEX instead of SIMPLE. The new regex (`/A new session was started via \/new or \/reset(?:\.\s*(?:Run|Execute) your Session Startup sequence|\s+while bootstrap is still pending)/`) is unanchored and accepts both verbs plus the PENDING/LIMITED continuation, restoring the `session_startup` override on 2026.4.27 while staying compatible with 2026.4.11. Lives in the new shared utility (see "Added"); the model-routing-specific override lives at [overrides.ts](src/middlewares/model-routing/scoring/overrides.ts) and the early-return that prevents tool-floor / structured-output-floor escalation lives at [scorer.ts:84-86](src/middlewares/model-routing/scoring/scorer.ts).
- Context Editing: first-message exclusion in `filterMessagesForICC` now matches the OpenClaw 2026.4.27 reset-prompt shapes. The previous `extractTextFromMessage(m).trim().startsWith('A new session was started via /new or /reset')` check at [index.ts](src/middlewares/context-editing/index.ts) broke when openclaw >= 2026.4.27 prepended the daily-memory prelude — the trimmed body started with `[Startup context loaded by runtime]` instead, so `startsWith` returned false and the synthetic startup turn was counted toward the message-threshold counter and fed into ICC input. Replaced with a call to the shared `isSessionStartupMessage(...)`, which finds the canonical phrase anywhere in the body and recognizes all three current prompt variants. The skip window (advance `startIndex` past the startup user message and any non-user messages that follow it) is unchanged — only the detection predicate was swapped.
- Dashboard (Model Routing): the Editing-profile dropdown rendered backslash-u-2014 escape sequences as literal text instead of em-dashes. The source had JSX option text written with JS-style Unicode escapes, but JSX text nodes are not JavaScript string literals — they don't process those escapes, so the backslash + `u2014` flowed through to the rendered DOM as a six-character string. The dropdown options and the description above them all displayed the visible escape sequence instead of an em-dash. Replaced every JS-escape-style em-dash in JSX text with the actual em-dash character so the dropdown now reads `Eco — Prefer cheaper models` etc. ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)).
- Dashboard (Model Routing): a stray `0` rendered above "Configured Providers" when both the Chat and ICC daily totals were 0 (typical state on a fresh enable). The Cost Sources Today section was wrapped in `{(todayBySource.chat || todayBySource.icc) && (...)}` which evaluated to `0 && (...)` → literal `0` when both were zero, and Preact rendered the number as text. Dropped the conditional entirely; the cards now always render so the section is discoverable from day one ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)).
- Dashboard (Model Routing): reverted the cost chart from bars back to a filled line. uPlot drew default point markers on top of the bars (visible as a small dot at each bar's left edge) and the chart looked frozen with sparse data — `DailyCost` aggregates per-day, so multiple requests on the same day all roll into one bar that grows in height rather than producing new bars. A single day's data made the bar mode misleading. Reverted both [ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx) (no longer passes `bars`) and [Chart.tsx](src/dashboard/components/Chart.tsx) (the `bars` prop and `uPlot.paths.bars` branch were removed entirely since nothing else uses them).
- Model Routing: `extractText()` now strips the OpenClaw envelope before scoring, so simple chat actually scores as simple. Two specific leaks were closed: (1) the chat-completion translator emits sender-metadata as a separate `role: "custom"` message in 2026.4.27, and the previous filter (`role !== 'system' && role !== 'developer'`) let those messages flow into the scoring concatenation — adding 100+ chars of `Sender (untrusted metadata):\n\`\`\`json\n{...}\n\`\`\`` per turn; (2) every user-role message body still carries the `[Day YYYY-MM-DD HH:MM TZ]` timestamp prefix (~27 chars) injected by `injectTimestamp()`, plus, on 2026.4.11, the sender block inlined into the user-role body as well. Net effect before the fix: a 5-char `"hello"` user message reached the scorer as a 140+ char blob, never tripping the `shortMessageChars` (50) threshold and consistently routing simple chat to `STANDARD` via the keyword scorer. After the fix: the role filter at [text-extractor.ts:55-59](src/middlewares/model-routing/scoring/text-extractor.ts) drops `role: "custom"` alongside `system`/`developer`, and every surviving message body runs through the new shared `stripOpenClawEnvelope(...)` (see Added). Result: `"hello"` → 5 chars → `short_message` override fires correctly. Note: `tool_floor` still applies on top, so chat with tools available continues to land STANDARD via the floor — that's a separate, intentional knob (configurable `toolFloorTier` is a future change).
- Context Editing: `extractCleanedTextForICC` now uses the shared `stripOpenClawEnvelope` instead of its own two-pattern inline regex. Strict superset of the previous coverage — adds handling for `Conversation info`, `Thread starter`, `Replied message`, `Forwarded message context`, and `Chat history since last reply` envelope blocks (only `Sender (untrusted metadata):` and the timestamp prefix were stripped before). Behavior on non-OpenClaw / envelope-free messages is unchanged (fast-path no-op).

## [1.0.1] - 2026-04-28

Patch release. No runtime behavior changes — refactors the source layout to dodge ClawHub's static-scan false positives and renames the npm publish workflow for clarity.

### Changed

- HITL: extracted the local-script reader (used to infer Gmail/Drive API signatures from shell-invoked scripts) out of `tool-interceptor.ts` into a new `script-content-loader.ts` module. ## [1.0.0] - 2026-04-27

Initial public release. Six middlewares for OpenClaw, published on npm as `@sapience-ai-corporation/openclaw-middleware-suite` with **one subpath per middleware**. Four govern and protect the action surface (HITL, Guardrail, PII Sanitizer, Tool Call Limit); two optimize the request itself (Context Editing, Model Routing). Everything runs in-process — no external service calls, zero telemetry, all state local.

### Added — Middlewares

- **HITL (Human-in-the-Loop)** — `ALLOW` / `DENY` / `ASK` policy per module × method with `allowPaths` / `denyPaths` glob support; destructive-action classifier (`HIGH` / `CATASTROPHIC`); irreversibility scorer (0–100); memory-risk forecaster; trust-rate limiter; **TOTP 2FA (RFC 6238)** for catastrophic actions; WhatsApp/Telegram/TTY approval queue with TTL expiry; ArgsHash enforcement on approval; protected modules cover FileSystem, Shell, Browser, Network, Gateway, Gmail, GoogleDrive, Memory, Process — plus shell-subcommand routing (`gog`, `gdrive`, `rclone`), gateway endpoint reclassification, and MCP tool-name mapping.
- **Context Editing** — Single-LLM-call ICC compaction (Priority Preservation, Conflict Resolution, typed Entity Locks); proactive triggers on token (default 80k) and/or message-count (default 50) thresholds; configurable `triggerMode` (`token` / `message` / `both`); user-configurable "keep N recent messages verbatim"; custom compaction prompt and custom compaction model; falls back to OpenClaw native overflow compaction when ICC fails; per-compaction JSONL audit trail; per-session cumulative token-savings stats; session-pruning toggle.
- **Model Routing** — 23-dimension complexity scorer (14 keyword + 9 structural dimensions, Aho-Corasick trie, density clustering, sigmoid k=8, 0.45 ambiguity threshold) feeding a 4-tier system (`SIMPLE` / `STANDARD` / `COMPLEX` / `REASONING`); session momentum (length-weighted blend of the last 5 decisions); per-session model pinning with three-strike escalation; capability-filtered fallback chain (max 5 per tier); request deduplication (SHA-256, 30s window) and deterministic response cache (LRU, 200 entries / 10 min, opt-in); native provider adapters for OpenAI, Anthropic, and Google with SSE streaming; auto Anthropic `cache_control` injection and Google `cachedContent` token passthrough; 4 routing profiles (`eco` / `auto` / `premium` / `agentic`); 6 hard overrides (reasoning keyword, short-message, tool-floor, large-context, structured-output floor, `/new` `/reset` → `SIMPLE`); daily cost alerts (`$5` warn / `$20` critical, configurable, 90-day ledger); config hot-reload via `fs.watchFile`; 4 plugin hooks (`onBeforeScore` / `onAfterScore` / `onBeforeForward` / `onAfterForward`).
- **Guardrail** — Three lifecycle surfaces: `before_tool_call`, `before_agent_start`, and `before_message_write`. The latter **rewrites the persisted transcript synchronously** so the LLM never sees pre-redacted content. Six detection engines (regex, prefix, heuristic, Shannon entropy ≥ 4.0, NFKC Unicode normalizer with homoglyph + zero-width handling, OpenAI Moderation API with async → sync cache bridge); seven guards (sensitive paths — 52 patterns + symlink resolution; egress control — 25-domain allowlist + private-IP / `169.254.169.254` SSRF block; destructive commands — 22 built-in patterns + custom regex; content moderation; role impersonation — 16 patterns including ChatML, Llama, fake `[SYSTEM]`, tool-output tags; canary tracker — SHA-256 ring buffer with leakback re-redaction; output metadata scrubber); 53 detection rules across the regex / prefix / heuristic engines; configurable `moderation.rewriteThreshold` (`MEDIUM` / `HIGH` / `CRITICAL`); dry-run / shadow mode; per-decision JSONL audit log.
- **PII Sanitizer** — Field-level DLP with 4 actions (`ALLOW` / `REDACT` / `ESCALATE` / `BLOCK`) and 4 severity tiers (`LOW` / `MEDIUM` / `HIGH` / `CRITICAL`); recursive deep scanning of nested objects, arrays, and stringified JSON; shell-argument parsing via `ShellParser`; pattern catalog covering email, phone, SSN, credit cards, IBAN, IPs, plus 17 cloud-provider / API-key prefixes and regexes (AWS, OpenAI, Anthropic, Hugging Face, Slack, GitHub, Stripe, SendGrid, npm, GCP, JWT, Bearer, DB connection strings, …); first-class `escalate` channel that `HitlMiddleware` consumes for human approval.
- **Tool Call Limit** — Dual-scope budgeting (session + request) evaluated on every call; soft + hard tiers (warn before block); per-module × per-method granularity; rolling 24-hour window with auto-reset; virtual-session-id mapping for sub-session tracking; out-of-process reset via `resetAt` marker (the same mechanism `sai limits reset` writes).

### Security

- Zero-trust posture: every action evaluated against policy; unknown actions default to `ASK`.
- Synchronous blocking: agents wait for approval before executing; `before_message_write` rewrites are applied to the persisted transcript on the same turn.
- Immutable JSONL audit trails per middleware (HITL decisions, Context Editing compactions, Model Routing routes, Guardrail detections, PII detections, Tool Call Limit counters).
- TOTP 2FA generated off-device — the agent never sees the approval code.
- ArgsHash verification on approval retry prevents parameter substitution between request and execution.
- Defense-in-depth: Guardrail, PII Sanitizer, and HITL evaluate each call independently in the same `before_tool_call` pipeline.
