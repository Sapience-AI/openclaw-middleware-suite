## [1.0.3] - 2026-05-05

### Added

- Dashboard (Context Editing): refresh button on the page header, mirroring Model Routing's. Top-right of the Overview / Config / Logs tab strip; re-fetches `config`, `stats`, and the compaction audit log via the same `loadAll()` fetch fan-out the page uses on initial mount. Spins via the existing `@keyframes sai-spin` while in flight, label switches to `Refreshingâ€¦`, button is disabled while either initial load or refresh is pending. Two new state variables (`refreshing`, the existing `loading`) gate the disabled state. ([ContextEditingPage.tsx](src/dashboard/pages/ContextEditingPage.tsx))
- Model Routing: tier scoring boundaries and override thresholds are now editable from the dashboard and exposed in the init wizard. Three boundaries (`simpleStandard`, `standardComplex`, `complexReasoning`) drive the score-to-tier cutoff and four override thresholds (`shortMessageChars`, `largeContextTokens`, `reasoningKeywordMin`, `structuredOutputMinTier`) tune the fast-path classification rules that bypass dimension scoring. **Backend**: new `overrideThresholds` field on `ModelRoutingPolicyData` ([ModelRoutingPolicyStore.ts](src/middlewares/model-routing/storage/ModelRoutingPolicyStore.ts)) â€” declared with an inline `{ shortMessageChars?, largeContextTokens?, reasoningKeywordMin?, structuredOutputMinTier? }` shape (matching the `boundaryOverrides` precedent) so the public API surface doesn't have to leak the internal `OverrideConfig` type. `buildConfig` ([index.ts](src/middlewares/model-routing/index.ts)) shallow-merges it onto `base.scoring.overrides` mirroring how `boundaryOverrides` already merges onto `boundaries`. Only fields the user has touched are persisted â€” defaults flow through for everything else, so future default tweaks reach un-customized installs. **CLI**: new `sai router config --set-override "<name> <value>"` ([cli/config.ts](src/middlewares/model-routing/cli/config.ts)) accepts the four override names with appropriate type coercion (numeric for the first three, normalized-uppercase Tier for `structuredOutputMinTier`). The `sai router config` status display now shows effective values plus a yellow `<- N` marker on rows the user has overridden, for both boundaries and overrides. **Init wizard**: two new optional steps in [cli/init.ts](src/middlewares/model-routing/cli/init.ts) gated behind `Customize tier scoring boundaries?` and `Customize override thresholds?` confirms â€” the wizard pre-fills defaults (or existing overrides on re-run), prompts for each value, and only persists values that differ from defaults. **Dashboard**: 7 new fields in the Config tab's form ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)) â€” three boundary number inputs (step 0.01) and four override inputs (three numbers + one tier dropdown) â€” keyed `_boundary_*` / `_override_*` and marshaled to the same `boundaryOverrides` / `overrideThresholds` PUT-body fields the wizard writes. The dashboard API endpoint `/api/routing/tiers` ([dashboard-api.ts](src/shared/server/dashboard-api.ts)) added `overrideThresholds` to its passthrough allowlist (boundaries were already there). The 7 fields share the form with the existing tier / fallback / pinning / cache fields and the `dense` modifier from the Config refactor.
- Model Routing: per-profile tier configurations. The store ([ModelRoutingPolicyData](src/middlewares/model-routing/storage/ModelRoutingPolicyStore.ts)) now carries `tierOverridesByProfile: Partial<Record<RoutingProfile, Partial<Record<Tier, TierModelConfig>>>>` â€” each of the three profiles (eco / premium / agentic) keeps its own tierâ†’model map independently. Editing one profile's tiers in the dashboard or via `sai router tiers --profile <p> --set "<TIER> <model>"` no longer overwrites the others' saved configuration. The runtime config gains a `ModelRoutingConfig.tiersByProfile` field; [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) resolves the request's profile, then looks up `tiersByProfile[profile]` for tierâ†’model resolution. Defaults flow from `PROFILE_CONFIGS` (static) at runtime; the init wizard precomputes `buildProfileFromDiscovered` defaults for each profile and persists them so every profile has an explicit configuration on a fresh install.
- Shared utility: extracted OpenClaw session-startup detection into [src/shared/session-detection.ts](src/shared/session-detection.ts) â€” a single `SESSION_STARTUP_REGEX` / `isSessionStartupMessage(text)` source of truth consumed by both Model Routing (for the `session_startup` scoring override) and Context Editing (for the first-message exclusion in `filterMessagesForICC`). The regex covers all three current OpenClaw reset-prompt shapes (BASE, BOOTSTRAP_PENDING, BOOTSTRAP_LIMITED) and is intentionally NOT anchored to start-of-string so it still matches when openclaw >= 2026.4.27 prepends a `[Startup context loaded by runtime]` daily-memory prelude to the same user-role body. Future OpenClaw prompt drift fixes one regex; both middlewares pick it up. New tests: [test/shared/session-detection.test.mjs](test/shared/session-detection.test.mjs) (13 pure-utility), [test/model-routing/session-startup-override.test.mjs](test/model-routing/session-startup-override.test.mjs) (7 `scoreRequest` integration), [test/context-editing/session-startup-filter.test.mjs](test/context-editing/session-startup-filter.test.mjs) (11 `filterMessagesForICC` integration).
- ICC extraction signaling end-to-end. New shared utility [src/shared/icc-detection.ts](src/shared/icc-detection.ts) defines `ICC_EXTRACTION_MARKER = '[SAI:ICC_EXTRACTION]'`, `isIccExtractionCall(text)`, and `stripIccMarker(text)` â€” the contract between Context Editing's compaction-extraction calls and Model Routing's scoring override. **CE side**: [ContextCurator.ts](src/middlewares/context-editing/ContextCurator.ts) `composeExtractionPrompt` prepends the marker only when the resolved target is `sai-router/*` (i.e., the call is actually going through MR); for direct providers (`anthropic/...`, `openai/...`) the marker is omitted to avoid LLM-prompt noise on calls MR will never see. **MR side**: a new `icc_extraction` branch in [overrides.ts](src/middlewares/model-routing/scoring/overrides.ts) `checkOverrides` runs **before** `session_startup`, returns `{ tier: 'SIMPLE', score: -0.5, confidence: 1.0, reason: 'icc_extraction' }`, and is added to the early-return list at [scorer.ts](src/middlewares/model-routing/scoring/scorer.ts) so `applyToolFloor` / `applyStructuredOutputFloor` cannot escalate the tier (the literal "schema" keyword in the ICC prompt would otherwise trip the structured-output floor). After scoring, [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) calls `stripIccMarker(...)` on every user-role `body.messages` entry (handles both `content: string` and `content: [{type: 'text', text}]` shapes) so the upstream LLM provider never sees the marker. Net effect: a user setting `agents.defaults.compaction.model = sai-router/eco` lands compaction calls on the eco profile's SIMPLE tier deterministically, regardless of the transcript content the scorer would otherwise see. Added a new value `'icc_extraction'` to the `ScoringReason` union in [types.ts](src/middlewares/model-routing/types.ts).
- Cost source attribution + per-source daily budgets. [cost-tracker.ts](src/middlewares/model-routing/storage/cost-tracker.ts) now accepts an optional `source: CostSource` (`'chat' | 'icc' | (string & {})`) on every `record()` call; events without a source default to `'chat'` so legacy callers stay attributed correctly. `DailyCost` gained a `bySource: Record<CostSource, SourceCostEntry>` map with per-source `costUsd` / `requestCount` and independent `warnFired` / `criticalFired` latches. `CostAlertConfig.budgets` is a new optional field of shape `Partial<Record<CostSource, { dailyWarn?: number; dailyCritical?: number }>>` â€” sources without an entry are still tracked but not alerted on individually; per-source alerts log with a tagged prefix (`WARNING [icc]: ...`) for grep-ability. [DEFAULT_COST_ALERT_CONFIG](src/middlewares/model-routing/storage/cost-tracker.ts) ships with `budgets: { icc: { dailyWarn: 1.0, dailyCritical: 5.0 } }` so a runaway compaction loop is visible without affecting normal chat alerting. [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) derives `costSource` from `scoringResult.reason === 'icc_extraction'` once after `FINAL_TIER` and threads it through all three `costTracker.record()` call sites (the SSE-converter completion handler, the fallback-executor stream callback, and the terminal cost record). Per-source alerts fire **independently** of the existing aggregate `warnThresholdUsd` / `criticalThresholdUsd` â€” a chat day can trip the aggregate warn while an `icc` day can simultaneously trip its own per-source warn on the same call. New tests in [test/model-routing/cost-tracker-source.test.mjs](test/model-routing/cost-tracker-source.test.mjs) cover bucketing, default-to-chat, per-source warn/critical firing, latching, and independence from aggregate alerts.
- Shared OpenClaw envelope stripper. New utility [src/shared/openclaw-envelope.ts](src/shared/openclaw-envelope.ts) exports `LEADING_TIMESTAMP_PREFIX_RE`, `INBOUND_META_SENTINELS`, and `stripOpenClawEnvelope(text)` â€” the contract for removing the AI-facing scaffolding OpenClaw prepends to every user turn. Two pieces of envelope are handled: the `[Day YYYY-MM-DD HH:MM TZ]` timestamp prefix (`injectTimestamp()` in openclaw's `gateway/server-methods/agent-timestamp.ts`) and any sentinel-delimited fenced-JSON metadata block (`buildInboundUserContextPrefix()` in openclaw's `auto-reply/reply/inbound-meta.ts`). The regex and sentinel list are kept verbatim from openclaw's official `stripInboundMetadata` in `auto-reply/reply/strip-inbound-meta.ts` so a future drift in envelope format breaks one regex in one place. The function is a zero-allocation no-op when neither pattern is present, so it adds no overhead to non-OpenClaw callers, and the strip is defensive â€” a sentinel header without a fenced-JSON body is treated as user content (no false strips on legit user text that happens to contain a sentinel string). Covers all six current sentinels: `Conversation info`, `Sender`, `Thread starter`, `Replied message`, `Forwarded message context`, `Chat history since last reply`. Both 2026.4.11 (envelope inlined into user-role body) and 2026.4.27 (sender block split into a separate `role: "custom"` chat-completion message) are handled. New tests in [test/shared/openclaw-envelope.test.mjs](test/shared/openclaw-envelope.test.mjs) lock both versions' shapes in as fixtures.
- Dashboard (Model Routing): three new surfaces on the page header / Overview tab. **Refresh button** â€” top-right of the page header, in line with the Overview / Config / Logs tab strip. Hoisted the data-loading from `useEffect` into a reusable `loadAll()` so the same fetch fan-out powers both initial mount and explicit refresh. Spins via a new `@keyframes sai-spin` in [components.css](src/dashboard/styles/components.css) while in flight, label switches to `Refreshingâ€¦`, button is disabled while either initial load or refresh is pending ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)). **Reason + Confidence columns** in "Recent Routing Decisions" â€” both fields are already on `RoutingAuditEntry` (the audit log writes them per request); surfacing them here brings the dashboard table in sync with the `Tier / Model / Score / Conf / Reason` columns the `sai router stats` CLI prints, so users can correlate routing decisions across both surfaces. **Cost Sources Today** section â€” two cards (Chat / ICC Compaction) under the cost chart, always rendered (including on fresh enable when both spends are 0) so the section is discoverable from day one. Each card shows today's spend, request count, the source's description, and a horizontal progress bar capped at the source's critical threshold (or warn if no critical), filled green / amber / red depending on which threshold the spend has crossed. "Warn $X / Critical $Y" labels under the bar; falls back to "No per-source budget configured." when `costAlerts.budgets[source]` is unset. Two new helpers (`computeTodayBySource`, `SourceCostCard`) live at the bottom of the page module.

### Changed

- Guardrail: consolidated the standalone Output Guardrail middleware into Guardrail proper. **Operator surface**: the `output-guardrail` master toggle in `plugin_config.middlewares` is gone â€” output scrubbing is now controlled exclusively by `guardrail.outputScrubber.enabled` in the Guardrail config tree, edited via the Guardrail dashboard's Output tab or `sai guardrail output toggle <enable|disable|dry-run>`. Activation requires Guardrail itself to be enabled (master toggle on), then the scrubber feature flag turned on â€” both default to `false` so a fresh install ships with neither active. The `output-guardrail` row was removed from the dashboard Overview middleware list, from `sai enable / disable` accepted names, from the `sai init` middleware registry, and from `configSchema.middlewares` in [openclaw.plugin.json](openclaw.plugin.json). `MiddlewareName` in [shared/storage/cleanup.ts](src/shared/storage/cleanup.ts) loses the `'output-guardrail'` member; legacy `OUTPUT_GUARDRAIL_DIR` / `OUTPUT_GUARDRAIL_CONFIG_FILE` / `STORE_KEY_OUTPUT_GUARDRAIL` paths were folded into Guardrail's cleanup spec so a guardrail-disable sweeps any legacy residue from older installs. **Runtime + programmatic alignment**: [GuardrailMiddleware.beforeMessageWrite](src/middlewares/guardrail/GuardrailMiddleware.ts) now runs the security write scanner and the output scrubber in sequence inside one method â€” the scrubber operates on the security stage's rewrite when one occurred (matching the prior plugin-runtime behavior where two separate `before_message_write` registrations chained), bails to `undefined` on non-assistant roles or when `outputScrubber.enabled` is off, and respects `dryRunMode` (logs the would-have-been scrub without touching the message). [plugin/index.ts](src/plugin/index.ts) collapses the previous two `tryOn(api, 'before_message_write', ...)` calls into one wrapper that forwards to `guardrail.beforeMessageWrite`, so the OpenClaw plugin runtime and programmatic `MiddlewareRegistry`-style consumers now execute identical code through the same method. New `scrubCount` counter exposed via `GuardrailMiddleware.getStatus().stats`. Audit log entries from the scrubber now tag `module: 'guardrail:output-scrubber'` (was `'output-guardrail'`); existing audit entries with the old tag remain readable. Tests in [test/integration/pipeline.test.mjs](test/integration/pipeline.test.mjs) (B2 / B3 / D4) migrated from `createOutputGuardrailHook()` to `guardrail.beforeMessageWrite()` â€” same behavior verified through the consolidated entry point. The `OutputGuardrailConfig` back-compat alias in [public/guardrail.ts](src/public/guardrail.ts) still re-exports `OutputScrubberConfig`; new code should prefer the latter name.
- Context Editing: rewrote the OpenClaw-hook pipeline around a single load-bearing hook (`before_model_resolve`) so the middleware works on openclaw 2026.4.27+ where the previous hooks are silently dropped or deprecated. **Root cause of the breakage on 2026.4.27**: the `agent_end` and `llm_output` hooks were moved to a new "conversation-access" permission gate ([openclaw registry.ts:1956-1977](https://github.com/openclaw/openclaw)) added in 2026.4.24 â€” non-bundled plugins (which sapience-ai-suite is) need an explicit `plugins.entries.<id>.hooks.allowConversationAccess: true` flag in `openclaw.json` for those hooks to register. Without the flag the registration is silently dropped (warn-level diagnostic). Additionally, `before_agent_start` is `@deprecated` per [openclaw hook-types.ts:803](https://github.com/openclaw/openclaw) â€” the replacement is `before_model_resolve` (early phase, fires BEFORE the gateway's SessionManager opens the JSONL â€” the only safe window to open our own SM and append a compaction entry without forking the DAG). **The new pipeline**: a single registration in [plugin/index.ts](src/plugin/index.ts) (`before_model_resolve`) replaces the four legacy registrations (`before_agent_start` + `before_prompt_build` + `agent_end` + `llm_output`). On every turn, [ContextEditingMiddleware.beforeModelResolve()](src/middlewares/context-editing/index.ts) opens its own `SessionManager` instance on the session JSONL, walks `getEntries()` to (a) count user messages for trigger evaluation, and (b) sum each persisted assistant message's `input + output` usage into the per-session `setAccumulatedUsage()` counter (replaces the push-based `llm_output` path; same provider-reported precision because the entries carry the same `message.usage` shape). If the threshold is met, ICC extraction runs and `requestCompaction()` fires inline â€” appending the compaction entry via `appendCompaction()`. When the hook returns, the gateway opens its own SM_A fresh, reads the just-written compaction entry, and the **same turn's** LLM call sees compacted history (matching the previous 4.11 `agent_end â†’ before_agent_start` two-hop in a single hop). Functional behavior is identical: with `messageThreshold: 2`, the 3rd message is the first to see compacted history. **Code that was removed**: `beforeAgentStart()`, `agentEnd()`, `llmOutput()` methods (~250 lines combined) along with the `pendingCompactions`, `beforeAgentStartCompactions`, and `mainAgentRunIds` state maps that supported the schedule-then-consume pattern; the `accumulateAssistantUsage()` push-based setter on `ContextEditingStats` (replaced by `setAccumulatedUsage()`); and the now-unused `AgentStartContext`, `AgentEndContext`, `LlmOutputContext` imports. **New surface**: `ModelResolveContext` and `ModelResolveResult` types in [src/types.ts](src/types.ts) and a corresponding `beforeModelResolve()` slot on the `Middleware` interface, plus `loadSessionManagerClass()` (lazy + cached) and `computeAssistantUsageFromEntries()` private helpers on the CE class. `before_model_resolve` has been ungated and stable since at least openclaw 2026.2.22 (well before our `peerDependencies.openclaw >= 2026.4.11` floor) so a single registration works on every supported version â€” no legacy fallback or feature-detect needed.
- Disable cleanup: middleware-specific cleanup logic is no longer hard-coded into `shared/storage/cleanup.ts`. The shared cleanup runner now drives a generic dispatch â€” every middleware contributes a declarative `{ files, dirs, storeKeys }` spec for the uniform file/dir/key-removal flow, and (optionally) a lazy-loaded extras handler in a new `EXTRA_CLEANUP_LOADERS` map for cleanup that can't be expressed declaratively. Model Routing's three quirks â€” stripping `sai-router/*` entries from `openclaw.json`, scrubbing per-agent `auth-profiles.json` rows, and zeroing the in-memory `stats` counter in [proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts) â€” moved out of [shared/storage/cleanup.ts](src/shared/storage/cleanup.ts) into a new [src/middlewares/model-routing/storage/disable-cleanup.ts](src/middlewares/model-routing/storage/disable-cleanup.ts) exporting `runDisableCleanup()`. `shared/cleanup.ts` lazy-imports it via the loader map so `shared/` retains zero static dependencies on middleware code. `cleanupMiddleware()` now has no `if (name === 'model-routing')` branches â€” pure data-driven dispatch. Future middlewares with non-declarative teardown register one entry in the loader map and one file in their own folder. Old `cleanOpenclaw: boolean` field on `CleanupSpec` is gone; the `cleanOpenclawConfig` / `cleanAgentAuthProfiles` helpers that lived in `shared/cleanup.ts` were ported verbatim into the new MR module.
- Model Routing: there is no user-facing "default profile" concept anymore. All three profiles are always exposed in the OpenClaw model picker; users select one per request rather than configuring a single global default. The dashboard's "Default Profile" dropdown was relabeled "Editing profile" and now scopes tier writes to the chosen profile only â€” submitting the form sends `{ profile, tierOverrides, sessionPinningEnabled, providerCacheEnabled }` to `PUT /api/routing/tiers`, which nests `tierOverrides` under `tierOverridesByProfile[profile]` ([dashboard-api.ts](src/shared/server/dashboard-api.ts), [ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)). Switching the editing profile remounts the form (`key={editingProfile}`) so tier dropdowns re-seed from that profile's saved values. The internal `defaultProfile` field on `ModelRoutingConfig` is preserved as the **absolute fallback** when a request can't be matched to a profile (no `sai-router/*` model id, no `X-Router-Profile` header) â€” programmatic consumers can still pass `mr.initialize({ defaultProfile: 'premium' })` to control that. The proxy's `resolveProfile` falls back to `'eco'` if `defaultProfile` is somehow not a valid profile ([proxy/handler.ts](src/middlewares/model-routing/proxy/handler.ts)).
- Model Routing: `sai router tiers` now requires `--profile <eco|premium|agentic>` to scope an edit. `sai router tiers --profile premium --set "COMPLEX claude-opus-4-6"` writes only the premium profile's COMPLEX tier; the eco and agentic profiles are untouched. Without `--profile` the command prints the tier mappings for all three profiles, marking each tier as `(configured)` or `(default)` ([cli/tiers.ts](src/middlewares/model-routing/cli/tiers.ts)).
- Model Routing: `sai init` interactive flow reworked. The wizard now skips the "select one profile" prompt and instead asks "Would you like to customize tier mappings for any profile?" When yes, the user picks any subset of `eco / premium / agentic` from a multiselect and walks through tier customization for each chosen profile. Profiles the user doesn't customize get `buildProfileFromDiscovered` defaults persisted explicitly so every profile slot exists in storage on a fresh install. The universal-fallback step now applies the chosen fallback across every tier of every profile ([cli/init.ts](src/middlewares/model-routing/cli/init.ts)).
- Model Routing: `buildConfig` ([index.ts](src/middlewares/model-routing/index.ts)) deep-clones `tiersByProfile` from the default before applying overrides so cross-call mutations don't leak into the module-level constant.
- Model Routing: renamed the four routing model display names from `"Auto (Smart Router â€” Balanced)"` / `"Eco (Smart Router â€” Cost Optimized)"` / `"Premium (Smart Router â€” Best Quality)"` / `"Agentic (Smart Router â€” Tool Optimized)"` to `"sai-router/auto"` / `"sai-router/eco"` / `"sai-router/premium"` / `"sai-router/agentic"` in [router-provider.ts](src/middlewares/model-routing/router-provider.ts). The OpenClaw `/model` UI dropdown renders `entry.name`, while `openclaw configure` labels options with the raw `provider/id` key â€” so the two surfaces showed different strings for the same routing models. Aligning `name` to the key form makes both surfaces match. Model `id`s are unchanged, so the routing proxy and tier resolution are unaffected. Existing users will pick up the new names automatically by toggling the middleware off and on in the dashboard (the disable cleanup strips `sai-router` entries from openclaw.json, and the re-enable persist-defaults handler runs the full init which re-injects fresh model definitions via `injectModelsConfig`).
- Plugin runtime: switched to the OpenClaw 2026.4.27 runtime config API (`runtime.config.current()` and `runtime.config.replaceConfigFile()`) as the sole supported path. With this release's peerDep floor at `>=2026.5.3`, both methods are guaranteed present, so [config-manager.ts](src/plugin/config-manager.ts) no longer feature-detects â€” `loadOpenClawConfig` calls `_runtime.config.current()` directly and `saveOpenClawConfig` calls `_runtime.config.replaceConfigFile({ nextConfig, afterWrite })` directly. The `OpenClawRuntime` interface in [plugin/index.ts](src/plugin/index.ts) now declares both methods as required (no longer optional), so any future regression that tries to reach for the dropped APIs is caught by TypeScript at the call site. Writes default to `afterWrite: { mode: "auto" }`; the `"auto"` policy delegates reload-vs-restart to the gateway's reload planner. `loadOpenClawConfig` / `saveOpenClawConfig` keep the same external signatures, so callers (Model Routing's `injectModelsConfig` / `cli/init.ts`, Context Editing's `cli/init.ts` / `pruning.ts` / `model.ts`, the disable-cleanup path in [shared/storage/cleanup.ts](src/shared/storage/cleanup.ts), `stageOpenClawWrites` / `flushToOpenClaw` in [shared/server/openclaw-sync.ts](src/shared/server/openclaw-sync.ts), and the toggle CLI in [shared/cli/toggle.ts](src/shared/cli/toggle.ts)) are unaffected.
- Context Editing: ICC extraction now honors `agents.defaults.compaction.model` and falls back through the agent primary model on failure. Both extraction paths in [ContextCurator.ts](src/middlewares/context-editing/ContextCurator.ts) (`extractViaLLM` and `extractViaCustomPrompt`) iterate a target list returned by the new `resolveExtractionTargets(api)` helper â€” `compaction.model` first, `model.primary` second, deduped â€” and try each in order until one succeeds. Previously only `model.primary` was read, which silently routed every compaction call through the user's chat model regardless of how the user had configured `sai context-editing model --set <model>` or the dashboard's compaction-model field. The cascade is now `compaction.model â†’ model.primary â†’ regex extraction`; a misconfigured compaction model (wrong id, bad API key, provider down) silently downgrades to the primary model instead of degrading all the way to regex. JSON-parse errors do **not** trigger fallback (they're upstream-format issues, not target-failure issues) and surface as `null` immediately. The marker decision is per-attempt â€” when the first target is `sai-router/*` and the second target is a direct provider, the marker is included on attempt 1 and dropped on attempt 2 so the direct-provider call doesn't carry unexplained noise. Each ICC call site went from ~14 lines of duplicated config-reading + slash-splitting + provider/model assembly to a single `resolveExtractionTargets(api)` call.

Universal Fallback / Session Pinning / Provider Prompt Caching stay un-indented because they're profile-level settings, not tier mappings ([ModelRoutingPage.tsx](src/dashboard/pages/ModelRoutingPage.tsx)).

Row 1 is **Routing Stats** (4 cards in a 2Ă—2 inner grid) on the left, **Cost Sources Today** (Chat / ICC Compaction cards) on the right. Row 2 is **Configured Models** (the previous `tier-grid` 4-up was switched to a 2Ă—2 inner grid since each row half is constrained to 50% width) on the left, **Configured Providers** table on the right. - Context Editing: trigger thresholds no longer fire spuriously the first time CE observes a session that already has messages. **Symptom**: enabling CE mid-session on a conversation with N pre-existing user messages (or any scenario where the in-memory `TriggerEvaluator` buffer was rebuilt empty against a JSONL that had already accumulated content â€” gateway restart with active session, plugin hot-reload, disableâ†’re-enable cycle) caused `beforeModelResolve` on the very next turn to compute a delta against `baselineMessageCount: 0`, so a 7-message session with `messageThreshold: 2` would compact immediately on the 8th message. **Root cause**: `getOrCreateBuffer` initializes a fresh `SessionBuffer` with `baselineMessageCount: 0` / `baselineTokens: 0` (correct for never-before-seen sessions starting from zero), but `syncSessionStats` then assigns the live counts to `messageCount` / `estimatedTokens` without re-anchoring the baseline â€” so `messageDelta = messageCount - 0 = messageCount` includes every historical message as if it were new. The post-compaction codepath (`resetSession`) handles this correctly by re-anchoring `baseline = messageCount`; the missing piece was an analogous anchor on the first `syncSessionStats`. **Fix**: added a `hasBeenSynced: boolean` flag to [SessionBuffer](src/middlewares/context-editing/types.ts), initialized `false` in [TriggerEvaluator.getOrCreateBuffer](src/middlewares/context-editing/TriggerEvaluator.ts). Fresh-session timing is unchanged because at Turn 1's `before_model_resolve` the JSONL has 0 user messages (the just-sent message hasn't been written yet by openclaw's SessionManager) â€” the anchor is trivially 0, and the delta-from-zero progression below proceeds as before.
